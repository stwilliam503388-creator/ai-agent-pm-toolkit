## 核心概念

## 1. 向量数据库为什么重要

RAG的检索环节离不开向量数据库。选错向量库 = 检索慢 + 成本高 + 扩展难。

## 2. 四大向量库对比

| 向量库 | 类型 | 部署 | 适合场景 | 不适合 |
|--------|------|------|---------|--------|
| Pinecone | 云托管 | SaaS | 快速原型、不想管运维 | 预算有限、数据敏感 |
| Milvus | 开源 | 自部署/云 | 大规模、高性能 | 小数据量（overkill） |
| pgvector | PG插件 | 自部署 | 已有PG、混合查询 | 纯向量（性能不如专用库） |
| Chroma | 开源 | 嵌入式 | 开发测试、小规模 | 生产大规模 |

## 3. 选型决策树

```
数据量 < 100万条？
  是 -> 有PostgreSQL？-> 是 -> pgvector
                      -> 否 -> Chroma
  否 -> 数据敏感不能上云？-> 是 -> Milvus自部署
                         -> 否 -> Pinecone
```

PM选型关注：不管运维选Pinecone（贵但省心），有DBA选Milvus（便宜但费人）。切换成本巨大，初期慎选。

## 实战选型：从数据量、预算、团队三个维度决策

## 实战选型：三个真实场景的决策

### 场景A：创业公司做MVP（数据量 < 10万条，预算紧张，团队2人）

选Chroma。理由：
- 嵌入式运行，不需要额外部署和维护
- 10万条数据量Chroma完全够用
- 零成本，零运维
- 后期扩展到100万条时再迁移到pgvector或Pinecone

风险：Chroma的社区不如Pinecone活跃，遇到坑可能搜不到解决方案。但你只有2个人，选Pinecone的账单会让你在MVP没验证前就烧完预算。

### 场景B：中型公司内部知识库（数据量 50万条，已有PostgreSQL，团队5人）

选pgvector。理由：
- 已有PostgreSQL基础设施和DBA团队
- 混用SQL+向量查询是核心需求（"查询最近一个月的产品相关文档"需要SQL过滤+语义搜索）
- 不需要额外服务，运维成本低

不选Pinecone：50万条数据的Pinecone月费不低，而且你已经为PG付了钱。
不选Milvus：50万条数据用Milvus是overkill，部署和维护成本远超pgvector。

### 场景C：大厂AI搜索产品（数据量 > 1亿条，需要毫秒级延迟，团队10人）

选Milvus。理由：
- 亿级数据的向量检索性能，Milvus业界最优
- 分布式架构支持横向扩展
- GPU加速索引构建
- 自部署满足数据安全合规要求

不选Pinecone：1亿条数据的Pinecone账单会让你每个月的成本报表变成红色。
不选pgvector：这个量级pgvector的向量检索性能跟不上。

## 常见误区

**误区1：选最流行的**
纠正：流行不等于适合。Pinecone在Hacker News上最火，但在中国创业公司的场景下，pgvector或Chroma实际更合适。

**误区2：选能力最强的**
纠正：能力最强（Milvus）意味着部署最复杂、运维最重。为10万条数据部署Milvus集群 = 用大炮打蚊子。

**误区3：选了就不能换**
纠正：可以换，但成本很高。所有向量要重新Embedding和写入新库。所以MVP阶段选轻量的（Chroma/pgvector），验证后再决定是否迁移。"先轻后重"比"一开始就上全套"风险小得多。

## 面试要点

**Q1：向量数据库和传统数据库区别？**
- 传统：精确匹配（WHERE name=张三）
- 向量：语义相似度（找和张三描述最像的100条）
- 不是替代，是互补。pgvector就是在传统DB上叠加向量能力

**Q2：PM选向量库最容易犯什么错？**
- 选最流行而不是最合适
- 忽略运维成本
- 不考虑扩展上限（明年翻10倍怎么办）
