---

## 核心概念

## 1. 为什么需要混合搜索

纯语义搜索(Dense)：不懂专有名词（搜BMW找不到宝马）
纯关键词搜索(Sparse)：不懂同义词（搜汽车找不到轿车）

混合搜索 = Dense(语义) + Sparse(关键词)，取两者之长。

## 2. 三种搜索对比

| 类型 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Dense | 向量相似度 | 理解语义 | 专有名词弱 |
| Sparse | BM25/TF-IDF | 精确匹配强 | 不懂同义词 |
| Hybrid | Dense+Sparse融合 | 两者之长 | 实现复杂 |

## 3. Reranker的作用

初检索(1000条->100条，快速粗筛) -> Reranker(100条->10条，精细排序)

Reranker比初检索模型更精确但更慢，只对Top-K结果排序。典型如BGE-Reranker、Cohere Rerank。

## 4. 典型混合搜索架构

```
用户查询 -> Dense检索(1000条) + Sparse检索(1000条)
  -> RRF融合去重(200条) -> Reranker重排序(10条) -> 注入Prompt
```

## 实战对比：纯语义 vs 混合搜索的真实差距

## 实战对比：同一查询在不同搜索策略下的表现

测试数据：5000篇中文技术文档，100个测试查询。分为三类查询：自然语言（如"怎么优化数据库性能"）、专有名词（如"Redis Cluster vs Sentinel"）、混合型（如"PostgreSQL 16的新特性对OLAP有什么提升"）。

| 搜索策略 | 自然语言 MRR | 专有名词 MRR | 混合型 MRR | P99延迟 |
|---------|------------|------------|-----------|--------|
| 纯Dense (bge-m3) | 0.89 | 0.62 | 0.78 | 35ms |
| 纯Sparse (BM25) | 0.45 | 0.91 | 0.55 | 10ms |
| Hybrid (RRF融合) | 0.88 | 0.87 | 0.86 | 45ms |
| Hybrid + Reranker | 0.92 | 0.90 | 0.90 | 120ms |

### 关键发现

1. **专有名词是纯语义搜索的死穴**。对"Redis Cluster vs Sentinel"这种查询，纯语义搜索的MRR只有0.62——因为"Cluster"和"Sentinel"在语义空间里可能跟各种运维概念混在一起，但Sparse搜索通过精确匹配能直接定位。

2. **大部分查询是混合型**。纯自然语言和纯专有名词查询各占少数，70%的真实查询是混合型。这意味着混合搜索不是nice-to-have，是必须的。

3. **Reranker的ROI有争议**。加了Reranker，MRR从0.86提升到0.90，但延迟从45ms涨到120ms（2.6倍）。如果你的产品对延迟敏感（如实时客服），Reranker可能弊大于利。如果对质量要求极高（如法律检索），120ms值得。

### PM决策框架

```
延迟要求 < 50ms -> Hybrid (无Reranker)
延迟要求 < 200ms 且 准确率要求高 -> Hybrid + Reranker
成本极度敏感 -> Dense only，在专有名词查询时兜底用关键词
```

## 常见误区

**误区1：有了Reranker就不用优化初检索**
纠正：Reranker只能排序，不能召回。初检索漏掉的关键文档，Reranker没有超能力找回来。优化初检索的召回率 > 优化Reranker的排序精度。

**误区2：RRF融合就是取平均**
纠正：Reciprocal Rank Fusion (RRF)不是简单平均。它的核心思想是让在任一系统中排名都很高的文档胜出，而不是让在一个系统中排名极高但在另一系统中没出现的文档胜出。这是有意设计的行为，不是妥协。

**误区3：混合搜索一定比单一搜索好**
纠正：在某些场景下，混合搜索可能不如单一搜索。比如用户搜索一个极其罕见的专业术语（如某个API函数名），Sparse搜索已经完美匹配了，混合搜索反而因为语义搜索召回了不相关的结果而降低了整体质量。

## 面试要点

**Q1：PM需要知道Reranker的技术细节吗？**
- 不需要知道怎么训练，但要知道：它比向量检索慢10-100倍、中文和英文Reranker不能混用、它是额外成本

**Q2：产品真的需要混合搜索吗？**
- 用户主要搜自然语言 -> Dense够
- 用户经常搜专有名词/ID -> 需要Sparse
- 检索结果经常不够精准 -> 需要Reranker
