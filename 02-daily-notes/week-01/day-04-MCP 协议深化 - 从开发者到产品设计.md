## 核心概念

## 1. MCP的核心价值：标准化工具发现与调用

MCP之前，每个Agent框架有自己的工具调用方式。LangChain有Tool类，AutoGPT有Plugin。MCP定义一个标准协议，让任何Agent用同样方式发现和调用任何工具。

PM视角类比：MCP之于AI工具 = USB-C之于硬件接口。

核心三要素：Tool（工具定义）、Resource（数据资源）、Prompt（提示词模板）。

## 2. Function Calling vs MCP vs Skill：三层架构

2026年Agent面试第一高频考点：

| 层级 | 概念 | 做什么 | 类比 |
|------|------|--------|------|
| 语言层 | Function Calling | 模型输出JSON表达调用意图 | 大脑说拿杯子 |
| 工具箱层 | MCP | 标准化工具发现与调用 | 手知道杯子在哪 |
| 操作手册层 | Skill | 封装完整业务流程 | 调酒SOP手册 |

三层不竞争，是分工：FC做底层通信 -> MCP做工具注册 -> Skill做业务封装。

## 3. MCP的PM决策清单

**用MCP**：跨框架兼容、工具多且变化频繁、需要权限控制

**不用MCP**：只有1-2个工具且不扩展、延迟极度敏感、封闭系统

**安全边界设计**：
- 工具分级：只读 vs 写入 vs 执行
- 用户确认：高风险操作（删除/支付）需二次确认
- 调用频率限制：防止工具滥用

## 实战案例：MCP Server的产品设计全过程

## 实战案例：从0设计一个MCP Server

假设你要为公司的内部工具做一个MCP Server，让Agent能查询Jira任务、创建Confluence文档、发送Slack消息。

### Step 1：工具清单设计

| 工具 | 功能 | 权限 | 风险 |
|------|------|------|------|
| jira_search | 搜索Jira任务 | 只读 | 低 |
| jira_get_issue | 获取单个任务详情 | 只读 | 低 |
| jira_create_issue | 创建任务 | 写入 | 中（可能创建重复任务） |
| confluence_search | 搜索文档 | 只读 | 低 |
| confluence_create_page | 创建页面 | 写入 | 中 |
| slack_send_message | 发送消息 | 执行 | 高（可能发错人/发错内容） |
| slack_list_channels | 列出频道 | 只读 | 低 |

### Step 2：权限分级决策

你最纠结的决策：slack_send_message该设为写入还是执行？

- 写入：Agent可以发消息，但需要用户确认。每次发消息弹确认框 = 体验差，用户可能关掉确认直接授权 = 等于没防护。
- 执行：Agent可以自主发消息，但有频率限制（每分钟最多5条）和内容过滤（禁止发送含链接的消息到公开频道）。

你选了"执行+护栏"方案。理由是：如果每次发消息都要确认，用户会把这个Agent关掉。用技术手段控制风险（频率限制、内容过滤、白名单频道），比用人肉确认更可持续。

这是典型的PM决策——不是"要不要安全"，而是"用什么方式做安全"。

### Step 3：错误处理设计

| 场景 | Agent看到什么 | 用户看到什么 |
|------|-------------|------------|
| Jira API超时 | "搜索超时，请缩小搜索范围" | "正在重试..." |
| 权限不足 | "你没有权限创建Jira任务" | "此操作需要管理员授权" |
| 重复创建 | "已存在相似任务[链接]，是否继续?" | 同上 |

关键设计：Agent看到的错误信息和用户看到的可以不一样。Agent需要技术细节来决策（重试?降级?放弃?），用户只需要知道"发生了什么+我该做什么"。

## 常见误区

**误区1：MCP就是加强版Function Calling**
纠正：Function Calling是模型能力（输出JSON），MCP是协议标准（定义工具）。FC让你能调用工具，MCP让你能管理工具生态。类比：FC是电话线，MCP是全套电话交换系统。

**误区2：所有工具都应该走MCP**
纠正：工具数量<3且永远不扩展的场景，直接用FC更轻量。MCP的价值在多工具管理和跨框架复用。过度设计也是错误。

**误区3：MCP Server就是CRUD包装器**
纠正：好的MCP Server不只是调API，是在API之上加产品逻辑。比如jira_create_issue不只是POST数据，还检查是否重复、自动分配负责人、添加默认标签。工具设计本身就是在做产品。

## 进阶思考：MCP生态的护城河

Anthropic推MCP不是做慈善。MCP的护城河是工具生态——一旦足够多的服务提供MCP Server，新Agent框架不兼容MCP就没工具可用。这就是"协议即平台"的玩法。

作为PM，理解这个层面的博弈会让你在架构讨论中有话语权。不只是"我们用MCP"，而是"我们用MCP是因为它在生态之战中领先，切换成本对我们有利/不利"。

## 面试要点

**Q1：怎么说服技术团队采用MCP而不是自建协议？**
- 生态优势：已有200+ MCP Server可复用
- 维护成本：社区维护 vs 自己维护
- 招聘优势：候选人更可能熟悉MCP
- 承认缺点：协议还在快速迭代，可能有breaking changes

**Q2：MCP和A2A有什么区别？**
- MCP：Agent<->工具（垂直），解决怎么调用工具
- A2A：Agent<->Agent（水平），解决怎么协作
- 不合并原因：场景不同。工具调用是单次同步的，Agent协作是持续异步的
