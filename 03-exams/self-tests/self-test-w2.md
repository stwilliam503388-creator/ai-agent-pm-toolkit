# 每周自测 #2 — Week 2

## Q1：概念解释（5分）
解释多模态Agent中"视觉理解"和"OCR"的本质区别，各举一个应用场景。

### 参考答案要点
- OCR：从图中提取文字（机械的，"图里写的是什么字"）
- 视觉理解：理解图中语义（"图中发生了什么"）
- OCR场景：扫描合同提取条款
- 视觉理解场景：分析UI截图判断操作是否正确
- OCR是基础能力，视觉理解是进阶能力

## Q2：概念解释（5分）
A2A协议中Agent Card的作用是什么？它与MCP的Tool定义有何不同？

### 参考答案要点
- Agent Card：描述Agent能做什么（能力声明），用于Agent间发现
- MCP Tool定义：描述工具的输入输出（功能声明），用于Agent调用工具
- 区别：Agent Card说"我能做研究"，Tool说"search(query)返回结果列表"
- 一个是能力承诺，一个是功能契约

## Q3：选型决策（8分）
你的Agent产品日均1000次调用，每次平均输入2000 tokens、输出500 tokens。你需要在GPT-4o（$2.50/$10.00 per 1M）和DeepSeek（$0.14/$0.28 per 1M）之间选择。请计算两者的日成本和月成本，并给出选型建议。

### 参考答案要点
- GPT-4o日成本：1000*(2000*2.50+500*10.00)/1M = $10/天，月$300
- DeepSeek日成本：1000*(2000*0.14+500*0.28)/1M ≈ $0.42/天，月$12.60
- 成本差25倍。但需要考虑：DeepSeek是否支持多模态？质量是否满足？
- 建议：核心场景用GPT-4o，非核心用DeepSeek，混合方案月成本可控制在$100左右

## Q4：选型决策（8分）
对比Firecrawl和baoyu-url-to-markdown，什么场景该选哪个？

### 参考答案要点
- Firecrawl：云端API、内置反反爬、付费。适合：不想管运维、需要规模化抓取
- baoyu：本地Chrome CDP、免费。适合：小规模、已有Chrome、数据敏感不能上云
- 选型判断：抓取量>1000条/天选Firecrawl；<100条选baoyu
- 迁移成本：从baoyu到Firecrawl的API改造工作量也要考虑

## Q5：产品设计（12分）
设计一个多模态Agent产品，帮助设计师从设计稿截图自动生成前端代码。你需要：分析这个场景需要哪些多模态能力、给出技术方案、评估难点。

### 参考答案要点
- 多模态能力：视觉理解（识别UI元素）+ 空间推理（布局关系）+ OCR（提取文字）
- 技术方案：截图->视觉模型(GPT-4V)->结构化描述->代码模型(Claude)->代码
- 评估难点：代码还原度（像素级差异）、响应式处理、组件库匹配
- 风险：视觉模型可能误识别元素类型；成本高（每张截图可能消耗上万tokens）
